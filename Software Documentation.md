## 1.3 Product Definition

**Smart Life Planner** is an **AI-powered personal life operating system** that combines productivity management, personal development, wellbeing support, and spiritual routine management into one intelligent platform.

Instead of forcing users to switch between separate apps for:

- task management
- note-taking
- habit tracking
- focus sessions
- journaling
- reminders
- prayer support

Smart Life Planner unifies these capabilities into a single intelligent ecosystem that helps users manage their lives more efficiently and with less mental overload.

The system is designed not as a passive to-do list, but as an **active AI-assisted life management platform** that helps the user capture, organize, schedule, reflect, and act.

## 1.7 Core Philosophy

The philosophical foundation of Smart Life Planner comes from two combined directions:

### A. AI-Native Productivity
The research foundation defines the product as a system that should actively absorb the “meta-work” of planning instead of forcing the user to manually organize everything. The goal is for the system to assist the user in execution and reflection, not just store information. 

### B. GTD (Getting Things Done)
Smart Life Planner is strongly inspired by the GTD methodology, which is built around the idea that the mind should not be used as a storage system for unfinished commitments. GTD emphasizes a trusted external system, next-action thinking, and a workflow that includes capture, clarify, organize, reflect, and engage.

Therefore, Smart Life Planner is built on this combined principle:

> **The app should help users think less about managing their system and more about actually living and doing meaningful work.**

- to-do apps for tasks
- note apps for ideas
- focus apps for Pomodoro
- habit apps for routines
- prayer apps for spiritual needs
- journaling apps for reflection

This fragmentation creates major problems:

- app switching fatigue
- poor integration between life areas
- maintenance overhead
- missed context between tasks, habits, and notes
- static reminder systems that users ignore
- stressful and unrealistic planning systems
- no unified connection between productivity and wellbeing
- no support for spiritual alignment within modern productivity systems 

Current tools also fail because many of them are either:

- too manual and require constant setup
- too aggressive and create planning anxiety
- too narrow and focus only on one life domain
- too passive and depend on the user to do all the thinking 

Smart Life Planner is designed to solve this by becoming a **single, intelligent, assistive life platform**.

## 1.13 Jobs-to-be-Done

Smart Life Planner is designed to fulfill these core user jobs:

### Job 1 — Clear mental overload
When the user feels mentally overloaded, the app must help capture and structure thoughts quickly so the working memory is cleared. 

### Job 2 — Tell the user what to do next
When the user is ready to work, the app should help identify the most relevant next task based on context, deadlines, priorities, habits, and available time. 

### Job 3 — Help users start difficult tasks
When the user procrastinates or feels blocked, the app should reduce emotional friction by helping them break tasks into smaller actions and restart momentum.

### Job 5 — Create one trusted system
When the user wants control, the app should function as the trusted external system that GTD requires, so important commitments are not held only in the mind.

## 1.15 GTD as the Behavioral Backbone

The behavioral workflow of Smart Life Planner follows the GTD model:

| GTD Stage | Meaning | Smart Life Planner Translation |
|---|---|---|
| Capture | collect everything that has attention | quick capture inbox, voice capture, notes |
| Clarify | decide what the input means | AI task understanding, task parsing |
| Organize | place items in the right buckets | inbox, tasks, projects, calendar, habits |
| Reflect | review and stay current | daily summaries, weekly review, dashboards |
| Engage | do the right next action | smart recommendations, focus mode, action prompts |

This structure is important because GTD emphasizes that unmanaged “open loops” keep occupying the mind, while clearly defined outcomes and next actions reduce stress and improve engagement.

# 6.3 Database Design Principles

The Smart Life Planner database shall follow these principles:

### 1. User-Owned Data
Almost all business data belongs to a specific authenticated user.

Examples:
- tasks
- notes
- habits
- journal entries
- prayer logs
- focus sessions

Each such record must include a reference to the owning user.

### 2. Referential Integrity
Relationships between tables must be enforced using foreign keys.

Examples:
- task belongs to a user
- subtask belongs to a task
- prayer log belongs to a user
- focus session may belong to a task

### 3. Auditable Timestamps
Tables should store lifecycle timestamps such as:

- created_at
- updated_at

Event-based tables may also store:
- completed_at
- started_at
- ended_at
- logged_at

These timestamps should use **timestamp with time zone** (`timestamptz`) for correctness across regions and time zones. :contentReference[oaicite:1]{index=1}

The schema must support:

- MVP release

# 6.9 Deletion Rules

Deletion behavior must preserve integrity.

## Recommended Rules

### users
Deleting a user should cascade or soft-delete all owned data depending on business decision.

### task_projects
If a project is deleted:
- linked tasks should either:
  - move to no project
  - or be deleted only if explicitly confirmed

Recommended MVP:
- set `project_id = NULL` on linked tasks

### tasks
If a task is deleted:
- delete its subtasks
- preserve linked focus sessions if analytics history matters, or nullify `task_id`

Recommended MVP:
- cascade delete subtasks
- set linked focus_sessions.task_id to NULL

### habits
If a habit is deleted:
- keep logs only if historical analytics are needed
- otherwise cascade delete

Recommended MVP:
- soft-delete habit using `is_active = FALSE`

## 7.7 HTTP Method Conventions
The API shall use HTTP methods according to standard semantics:

* **GET** → retrieve resources
* **POST** → create resources or trigger non-idempotent actions
* **PATCH** → partially update resources
* **PUT** → fully replace a resource when needed
* **DELETE** → delete a resource 

**Practical Rule for This Project**
For Smart Life Planner:
* use **GET** for reads
* use **POST** for create actions and action-style routes
* use **PATCH** for partial updates like marking complete
* use **DELETE** for delete operations
* use **PUT** only where full replacement makes sense

# Step 8: AI System Design

## 8.1 Purpose

This section defines the **AI system design** of Smart Life Planner.

The AI layer is one of the core differentiators of the project.  
It transforms the app from a normal productivity application into an **AI-powered personal life operating system**.

The purpose of the AI system is to help users:
* capture tasks naturally
* reduce planning friction
* receive smart suggestions
* organize their day better
* improve productivity habits
* connect productivity with spiritual balance

This section defines:
* AI goals
* AI modules
* AI architecture
* AI data flow
* AI input and output behavior
* AI safety and validation rules
* MVP AI scope
* future AI expansion plan

---

## 8.2 AI Philosophy

The Smart Life Planner AI is designed as an **assistive intelligence layer**, not a replacement for user control.

The AI should:
* reduce friction
* suggest, not force
* clarify ambiguity
* personalize over time
* remain transparent
* stay aligned with the user’s goals and values

The AI must not behave like an uncontrolled autonomous agent.

Instead, it should act as:

> **A helpful personal life assistant that supports decision-making, structure, and consistency.**

---

## 8.3 Core AI Objectives

The AI system must help solve the following user problems:

### 1. Task Capture Friction
Users often think naturally, not in structured task fields.

AI should convert natural language into structured records.

Example:

```text
Finish database assignment tomorrow at 8 PM

Converted into:

Title: Finish database assignment
Due Time: Tomorrow 8:00 PM
Priority: Medium
Estimated Duration: Optional
```

### 2. Planning Overload
Users often know what they need to do, but struggle with:
* prioritization
* sequencing
* deciding what to do next

AI should provide:
* next-task suggestions
* daily schedule assistance
* task prioritization help

### 3. Inconsistent Productivity
Users often fail to maintain habits and focus routines.
AI should provide:
* reminders
* behavior insights
* motivational nudges
* consistency analysis

### 4. Fragmented Daily Life
Users often separate:
* work
* study
* habits
* reflection
* prayer

AI should support a more connected life model by considering:
* deadlines
* work sessions
* habits
* prayer times
* personal routine

## 8.4 AI Design Principles
The AI system shall follow these design principles.

### 8.4.1 Human-in-the-Loop
The AI may suggest actions, but the user must remain in control.
Examples:
* AI suggests a task structure → user confirms or edits it
* AI suggests a daily plan → user accepts or ignores it

### 8.4.2 Structured Outputs
AI outputs should be converted into structured JSON-style data whenever possible.
This is important because AI results must be:
* saved in the database
* validated
* shown consistently in UI
* reused by backend logic

### 8.4.3 Domain Awareness
The AI should understand the domains inside Smart Life Planner:
* tasks
* notes
* habits
* focus sessions
* prayer times
* journal reflections

This means it must classify and route user input correctly.

### 8.4.4 Graceful Uncertainty Handling
If the AI is uncertain, it should:
* ask for clarification
* return partial results
* avoid inventing false details

Example:
If the user says:
```text
Meeting tomorrow
```

AI may know:
* title = Meeting
* date = tomorrow

But if no time is provided, the system should either:
* leave time empty
* ask user to confirm

### 8.4.5 Safety and Respectful Boundaries
The AI must:
* avoid harmful advice
* avoid pretending to know private facts it does not know
* avoid making irreversible changes without confirmation
* avoid confidently guessing sensitive data

## 8.6 AI Architecture Overview
The AI system follows a layered architecture.

**AI Processing Flow**
```text

   ↓
Input Source Layer

   ↓

   ↓

   ↓
```

**Responsibilities**

## 10.10 Timezone & Time Handling Rules
This subsystem is highly time-sensitive, so timezone handling must be explicit.
PostgreSQL’s current documentation recommends time-zone-aware date/time types for timezone use cases, stores timezone-aware values internally in UTC, and converts them for display according to session timezone. The PostgreSQL wiki also strongly recommends `timestamptz` instead of `timestamp without time zone` for real moments in time. 

**Rules**
* **Rule 1:** Store real-world event timestamps using `TIMESTAMPTZ`.
* **Rule 2:** Store each user’s timezone in profile/settings.
* **Rule 3:** Always compute reminder times relative to the user’s timezone.
* **Rule 4:** Avoid using time-only timezone fields for real-world moments.

PostgreSQL also does not recommend time with time zone for these use cases.

## 11.5 Authentication Design

Smart Life Planner will use **token-based authentication** centered around JWT bearer tokens, following FastAPI’s standard OAuth2 password flow pattern with secure password hashing and bearer-token authorization. FastAPI’s official security tutorial explicitly recommends password hashing and bearer JWTs, and notes that JWTs are signed rather than encrypted, so sensitive claims should not be placed inside them.

### 11.5.1 Authentication Flow

The authentication flow is:

```text
User submits email + password
        ↓
Backend validates credentials
        ↓
Password hash is checked securely
        ↓
Backend issues JWT access token
        ↓
Client stores token securely
        ↓
Client sends Bearer token on future requests
```
FastAPI’s official guidance for OAuth2 with JWT follows this same pattern and emphasizes that plaintext passwords should not be stored. 

### 11.5.2 Login Credentials
For MVP, Smart Life Planner supports:
* email
* password

Future optional methods:
* Google sign-in
* other trusted identity providers

Passwords must never be stored in plaintext. Instead, only password hashes are stored in the database. FastAPI’s official JWT security tutorial explicitly demonstrates password hashing rather than storing plaintext secrets. 

### 11.5.3 JWT Token Rules
JWT tokens shall be used as access tokens.

**Token Rules**
* include user identity reference
* include token expiration
* be signed securely
* be validated on every protected request

**Important Rule**
JWT contents must not include highly sensitive private data, because JWTs are signed but not encrypted by default. FastAPI’s security docs explicitly state that JWTs are not encrypted and can be decoded by anyone who has them. 

### 11.5.4 Session Expiration
The backend shall enforce token expiration.

**Rules:**
* expired tokens must be rejected
* protected endpoints must require valid token
* logout should invalidate access where revocation/blacklisting is implemented
* future refresh-token support may be added later

FastAPI’s JWT approach relies on expiring bearer tokens and validating them on protected routes. 

## 11.6 Authorization Design
Authentication proves who the user is. Authorization determines what that user is allowed to access.

Smart Life Planner is a user-owned data system, so authorization must be enforced on every user-scoped resource. PostgreSQL’s row-level security is specifically designed to restrict which rows a user can read or modify on a per-user basis. 

### 11.6.1 Ownership Rule
Every user-owned resource must satisfy:

```text
resource.user_id == current_user.id
```

**Examples:**
* task belongs to current user
* note belongs to current user
* habit belongs to current user
* prayer log belongs to current user
* journal entry belongs to current user

If ownership fails:
* return 404 Not Found or 403 Forbidden according to policy

### 11.6.2 API-Level Authorization
Every protected API endpoint must:
* authenticate the user
* resolve the requested resource
* verify resource ownership
* deny access if ownership fails

This rule applies to:
* GET
* PATCH
* DELETE
* action endpoints such as complete/reopen/log

### 11.6.3 Database-Level Isolation
Application-level authorization is required, but database-level protection should also be considered.

**Recommended database protections:**
* separate database role for application access
* avoid superuser use in normal app runtime
* use restricted privileges
* evaluate Row-Level Security (RLS) for highly sensitive user-owned tables

PostgreSQL row security can enforce per-row visibility and modification rules; when enabled and no policy exists, access is denied by default. PostgreSQL also notes that roles with BYPASSRLS can bypass such policies, so operational roles should be designed carefully.

* never return passwords in API responses
* never log passwords
* validate minimum password strength at registration/reset

FastAPI’s official JWT example uses password hashing and explicitly shows that only hashed passwords are stored.

## 12.6 Docker Compose Design
For local development and possibly staging, the project should use Docker Compose.

### 12.6.2 Compose Networks
Docker Compose sets up a default network for the app, and containers on that network can reach each other by service name. Compose also allows explicitly defined named networks, including isolated internal networks. 

**Recommendation**

* optional public-facing path only through proxy/backend as needed

### 12.6.3 Compose Volumes

**Recommended Volumes**

**Why**
* preserve database across container restarts

## 12.14 Backup and Restore Strategy
The database must support backup and restore.
PostgreSQL recommends `pg_dump` for backups and `psql` or `pg_restore` for restore depending on dump format. Text dumps are restored with `psql`, while non-text archive formats are restored with `pg_restore`. PostgreSQL also notes that the target database is not created automatically by a plain SQL restore. 

### 12.14.1 Backup Method
Recommended MVP backup method:
* scheduled PostgreSQL dumps
* stored in secure backup location

**Tools**
* `pg_dump`
* optional backup container/job

### 12.14.2 Restore Method
Restore options:
* plain SQL dump → `psql`
* non-plain archive dump → `pg_restore`

PostgreSQL documentation explicitly distinguishes these restore paths and notes that the destination database must already exist for plain SQL restore. 

### 12.14.3 Backup Rules
* backups must be periodic
* backups must be tested occasionally
* production restore procedure must be documented
* backup storage must not depend only on the live DB volume
